## 2. 클로드 코드 기본 사용법

### 2-1. 터미널 기초
- VS Code에서 터미널 여는 법
- `claude` 명령어로 클로드 코드 실행
- 클로드 코드 종료하는 법

> **의도**: 가장 기본적인 진입점. 터미널 공포증 해소

### 2-2. 핵심 명령어
- `/init` - CLAUDE.md 파일 생성, 프로젝트 컨텍스트 설정
  - "이 프로젝트가 뭔지 클로드에게 알려주는 파일"
- `/compact` - 대화 내용 압축, 토큰 한도 절약
  - "대화가 길어지면 클로드가 앞 내용을 까먹을 수 있어서 정리해주는 것"

> **의도**: Pro 플랜 한도 관리 습관 형성. 이후 실습에서 자연스럽게 사용하도록

### 2-3. 한도 아끼는 습관
- 요청 전에 생각 먼저 정리하기
- 한 번에 여러 기능 요청하지 말고 하나씩
- `/compact` 주기적으로 사용

> **의도**: Pro 플랜 한도 걸리면 해커톤 때 곤란. 미리 습관 들이기

## 3. 가계부 실습 - 시연

### 3-1. 프로젝트 시작
- 새 폴더 생성
- 터미널에서 클로드 코드 실행

> **의도**: 프로젝트 시작하는 기본 루틴 보여주기

### 3-2. Plan 모드와 프롬프트 잘 쓰는 법
- "가계부 웹앱 만들어줘" 요청
- 클로드가 플랜(계획)을 먼저 보여줌
- 플랜 검토하는 법
  - "이 구조가 맞는지 확인"
  - "빠진 기능 없는지 체크"
- 플랜 수정 요청하는 법
  - "여기에 OO 기능도 추가해줘"
  - "이 부분은 빼줘"
- 플랜 승인 후 개발 진행

> **의도**: 바이브 코딩의 핵심 = 플랜을 잘 짜는 것. 무작정 시키지 말고 계획부터.

### 3-3. 좋은 프롬프트 vs 나쁜 프롬프트
- 나쁜 예: "가계부 만들어줘"
- 좋은 예: "수입/지출 입력할 수 있고, 월별로 합계 보여주고, 카테고리별 분류되는 가계부 웹앱 만들어줘"
- 구체적으로 요청할수록 원하는 결과가 나온다

> **의도**: 프롬프트 품질이 결과물 품질을 결정한다는 것 체감

### 3-4. Edit 모드와 대화 관리
- 개발 진행 중 부분 수정 요청
  - "금액에 천 단위 콤마 추가해줘"
  - "색상을 파란색으로 바꿔줘"
- 대화창 관리
  - 이전 상태로 돌아가기
  - 잘못된 방향으로 갔을 때 되돌리는 법

> **의도**: 완벽하게 한 번에 안 돼도 괜찮다. 수정하면서 발전시키는 게 바이브 코딩

### 3-5. 완성 및 확인
- 로컬에서 실행
- 브라우저에서 가계부 작동 확인

> **의도**: "내가 만든 게 진짜 돌아간다!" 첫 성공 경험

## 5. Supabase 연결 - 시연

### 5-1. 왜 DB가 필요한가
- 지금 만든 가계부의 문제점 시연
  - 데이터 입력 후 새로고침 → 다 사라짐
- "브라우저를 꺼도 데이터가 남아있으려면 저장소가 필요하다"

> **의도**: DB의 필요성을 문제 상황에서 자연스럽게 이해

### 5-2. DB 개념 간략 설명
- 데이터베이스 = 데이터 저장소
- 테이블 = 엑셀 시트
- row = 한 줄 데이터
- Supabase = 쉽게 쓸 수 있는 클라우드 DB 서비스

> **의도**: 비개발자도 이해할 수 있는 수준으로 개념만 잡기. 깊게 안 들어감

### 5-3. Supabase 연결 과정
- Supabase 사이트에서 프로젝트 생성
- 테이블 만들기 (가계부용)
- 클로드 코드에게 "이 가계부에 Supabase 연결해줘" 요청
- API 키, URL 등 설정
- 새로고침해도 데이터 유지되는 것 확인

> **의도**: 클로드 코드가 DB 연결까지 해준다는 것 보여주기

## 7. 당근마켓 클론 - 시연

### 7-1. 왜 Git이 필요한가
- 지금까지는 혼자 작은 프로젝트
- 이제 더 큰 프로젝트, 여러 기능, 나중에 팀 협업
- 버전 관리의 필요성
  - "어제 잘 되던 게 오늘 안 되면?"
  - "이전 상태로 돌아가고 싶으면?"
- Git = 코드의 타임머신

> **의도**: Git을 왜 배워야 하는지 동기 부여

### 7-2. Git 설치 확인
- 터미널에서 `git --version` 확인
- 안 된 사람 트러블슈팅

### 7-3. 프로젝트 셋업
- 새 프로젝트 폴더 생성
- `/init` 실행
- `.claude.md` 파일 설명
  - "이 파일에 프로젝트 정보를 적어두면 클로드가 맥락을 이해한다"
  - "매번 설명 안 해도 된다"
- GitHub에 새 레포 생성
- 로컬과 GitHub 연결

> **의도**: 실무 프로젝트 시작하는 루틴 보여주기

### 7-4. 플랜 모드로 설계
- "당근마켓 클론 만들어줘. 상품 목록 페이지랑 상품 등록 페이지" 요청
- 플랜 검토
  - 전체 구조 확인
  - 페이지 구성 확인
  - 필요한 기능 확인
- 플랜 수정 및 승인

> **의도**: 큰 프로젝트일수록 플랜이 중요하다는 것 강조

### 7-5. 기능별 개발 + 커밋
- **기능 1: 상품 목록 페이지**
  - 구현
  - 확인
  - "지금까지 작업 커밋해줘" → 클로드가 git commit 실행
  - 커밋 메시지 확인
- **기능 2: 상품 등록 페이지**
  - 구현
  - 확인
  - 커밋
- **중간 푸시**
  - "GitHub에 푸시해줘"
  - GitHub에서 코드 올라간 것 확인

> **의도**: "기능 하나 완성되면 커밋" 습관. 나중에 문제 생겨도 돌아갈 수 있다

## 8. 당근마켓 - Git 브랜치 실습

### 8-1. 브랜치 개념 설명
- main 브랜치 = 원본, 안정된 버전
- feature 브랜치 = 새 기능 개발하는 작업 공간
- 각자 브랜치에서 작업 → 나중에 합칠 수 있음 (머지)
- 그림으로 설명

> **의도**: 협업의 기본 = 브랜치. 개념만 먼저 이해

### 8-2. 수강생 브랜치 작업

**레포 가져오기**
- 내(강사) 레포 URL 공유
- 각자 clone
  - "이 레포 클론해줘: [URL]"

**브랜치 생성**
- 각자 브랜치 생성
  - "feature/홍길동-수정기능 브랜치 만들어서 체크아웃해줘"
- 본인 브랜치로 이동된 것 확인

**기능 추가 개발**
- 과제: 상품 수정 기능 또는 상품 삭제 기능 만들기
- 각자 클로드 코드에게 요청해서 개발
- 완성되면 커밋
  - "지금까지 작업 커밋해줘"

**푸시**
- "GitHub에 푸시해줘"
- GitHub에서 본인 브랜치 올라간 것 확인

> **의도**: clone → 브랜치 → 개발 → 커밋 → 푸시 사이클 직접 경험. 협업의 기본 흐름

## 9. Vercel 배포 - 시연

### 9-1. 배포란?
- 지금까지는 내 컴퓨터에서만 실행 (localhost)

- Vercel = 쉽게 배포할 수 있는 서비스

> **의도**: 배포의 개념과 필요성

### 9-2. main 브랜치 배포
- Vercel 사이트에서 GitHub 레포 연결
- main 브랜치 배포
- 생성된 URL로 접속해서 확인
- "이제 인터넷 되는 곳이면 어디서든 접속 가능"

> **의도**: 배포가 이렇게 쉽다는 것 보여주기

### 9-3. 브랜치별 Preview 배포

- 수강생들 브랜치 푸시하면 각각 URL 생김
- 각자 본인 브랜치 URL 접속
- 본인이 만든 기능(수정/삭제) 작동하는 것 확인

> **의도**: "내가 만든 기능이 진짜 인터넷에 떠있다!" 성취감

## 1. 간단 챗봇 만들기

### 1-1. API 개념 설명
- API란?
  - "프로그램끼리 대화하는 방법"
  - "우리 앱이 Gemini한테 질문하고 답변 받는 통로"
- API 키란?
  - "Gemini를 쓸 수 있는 열쇠"
  - "이 열쇠가 있어야 Gemini가 응답해줌"

> **의도**: API 개념을 비개발자 눈높이에서 설명

### 1-2. 최신 문서 참고하는 법

**문제 상황 시연**
- 클로드 코드에게 "Gemini API로 챗봇 만들어줘" 요청
- 클로드가 모르거나 옛날 방식으로 하려고 함
- "클로드 코드가 학습한 이후에 나온 API라서 모를 수 있어요"

**해결 방법 시연**
- Gemini 공식 문서 URL 찾기
- "이 문서 참고해서 다시 해줘: [URL]" 요청
- 클로드가 문서 읽고 최신 방식으로 구현

**핵심 메시지**
- "AI 도구가 모든 걸 아는 게 아니에요"
- "최신 기술 쓸 때는 공식 문서를 같이 줘야 해요"
- "이게 바이브 코딩 실력 차이를 만들어요"

> **의도**: 실무에서 진짜 중요한 스킬. 클로드 코드의 한계와 극복법

### 1-3. 챗봇 개발 시연
- 새 프로젝트 폴더 생성
- Gemini API 문서 URL 제공
- "이 문서 참고해서 간단한 질의응답 챗봇 만들어줘" 요청
- 개발 진행
- 완성 후 테스트
  - "SKT 요금제 추천해줘" → 일반적인 답변

> **의도**: API 연동 웹앱 개발 흐름 보여주기

### 1-4. 각자 챗봇 개발
- 각자 새 프로젝트에서 시작
- GitHub 레포 생성
- 챗봇 개발
- 커밋 & 푸시
- Vercel 배포
- 배포된 URL에서 작동 확인

> **의도**: API 연동 + 배포까지 전체 사이클 직접 경험

## 2. 파일 기반 챗봇 (심화)

### 2-1. 파일 기반 답변 개념
- 아까 만든 챗봇의 한계
  - "SKT 요금제 추천해줘" → 일반적인, 정확하지 않은 답변
- 우리가 원하는 것
  - 실제 요금제 자료를 기반으로 정확한 추천
- 파일 첨부 답변
  - PDF, 엑셀 등 자료를 AI가 읽고 그 내용 기반으로 답변

> **의도**: 왜 파일 기반 답변이 필요한지 문제에서 출발

### 2-2. 개발 시연
- 아까 만든 챗봇 프로젝트 열기
- Gemini 파일 업로드 API 문서 URL 제공
- "이 문서 참고해서, 파일 업로드하면 그 파일 내용 기반으로 답변하는 기능 추가해줘"
- 개발 진행
- 요금제 자료(PDF/엑셀) 업로드 테스트
- "내가 영상 많이 보는데 요금제 추천해줘" → 자료 기반 정확한 답변

> **의도**: 기존 프로젝트에 기능 추가하는 흐름

### 2-3. 각자 심화 챗봇 개발
- 각자 본인 챗봇에 파일 기반 기능 추가
- 커밋 & 푸시
- 아까 배포한 URL에서 새 기능 반영 확인
  - "푸시하면 Vercel이 자동으로 다시 배포해요"

> **의도**: 기능 추가 → 푸시 → 자동 배포 사이클 경험. 실무와 동일한 흐름

## 3. 스토리→이미지 생성기 시연 (종합)

### 3-1. 완성본 시연
